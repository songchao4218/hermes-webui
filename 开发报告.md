## 一、项目概述

Hermes WebUI 是一个为 Hermes AI Agent 提供网页界面的开源项目。它作为 Hermes CLI 的图形化前端，允许用户通过浏览器与本地 AI Agent 交互，执行文件操作、代码执行、浏览器控制等任务。

### 核心价值主张

1. **共享记忆系统**: WebUI 与 Hermes CLI 读写相同的记忆文件，实现无缝切换
2. **100% 本地运行**: 所有数据存储在用户本地机器，无需云端服务
3. **身份定制**: 支持自定义 Agent 名称、头像、主题色
4. **智能路由**: 自动识别用户意图，在普通对话和 Agent 模式间切换

## 三、SOUL.md 与 Chat 模式关联问题分析

### 3.1 问题描述

用户在 SOUL.md 中写入的信息在 Chat 模式中未被识别。

### 3.2 原因分析

经过代码审查，发现以下关键流程：

**1. SOUL.md 读取流程** (`backend/hermes_bridge.py:119-144`):
```python
def build_system_prompt(self) -> str:
    """Build the system prompt by combining all memory files."""
    memories = self.get_all_memories()
    parts = []
    if memories["SOUL.md"]:
        parts.append(memories["SOUL.md"])
    # ... 其他记忆文件
    return "\n\n---\n\n".join(parts)
```

**2. Chat 端点使用记忆** (`backend/app.py:864-875`):
```python
# Build system prompt from Hermes memories
system_prompt = bridge.build_system_prompt()

# Build message history for context
messages = []
if system_prompt:
    messages.append({"role": "system", "content": system_prompt})
```

**3. 关键发现**:

✅ **普通 Chat 模式 (`/api/chat` 和 `/api/chat/stream`) 确实使用了 SOUL.md**
- `build_system_prompt()` 方法将 SOUL.md 内容作为 system prompt 注入
- 这部分内容会随每次请求发送给 Ollama

⚠️ **Agent 模式 (`/api/agent/run`) 不直接使用 SOUL.md**
- Agent 模式通过 `wsl.exe hermes chat` 调用 Hermes CLI
- Hermes CLI 有自己的记忆读取机制
- WebUI 只负责传递用户消息，不干预 Hermes 的系统提示词构建

### 3.3 问题诊断结论

**SOUL.md 在 Chat 模式中应该被识别**，如果未被识别，可能原因：

1. **文件路径问题**: SOUL.md 不在预期的 `~/.hermes/SOUL.md` 位置
2. **文件权限问题**: WebUI 无法读取该文件
3. **Ollama 模型问题**: 某些模型对 system prompt 的处理不佳
4. **内容格式问题**: SOUL.md 内容格式可能不符合预期

**建议排查步骤**:
1. 确认 `~/.hermes/SOUL.md` 文件存在且可读
2. 检查 WebUI 启动日志中的 "Hermes home" 路径是否正确
3. 通过 `/api/memories` 端点验证 SOUL.md 内容是否被正确读取
4. 检查 Ollama 日志确认 system prompt 是否被发送

## 五、代码质量评估

### 5.1 优势

1. **清晰的架构设计**: 前后端分离，职责明确
2. **完善的错误处理**: 大部分边界情况都有处理
3. **安全意识**: Token 认证、路径穿越防护、输入校验
4. **用户体验**: 流式响应、加载状态、错误提示
5. **跨平台支持**: Windows/WSL2、macOS、Linux

### 5.2 改进空间

1. **代码组织**:
   - `backend/app.py` 过长，建议按功能拆分为多个路由模块
   - `frontend/index.html` 单文件过大，建议模块化

2. **状态管理**:
   - `conversations` 全局字典在多用户场景下会冲突
   - 建议引入用户会话隔离或数据库持久化

3. **配置管理**:
   - 配置分散在环境变量、配置文件、代码默认值中
   - 建议统一配置管理方案

4. **日志记录**:
   - 部分关键流程缺少日志
   - 建议增加结构化日志
